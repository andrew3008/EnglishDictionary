## 1. Резюме для принятия решений

Проведена первая полная макро-сессия верификации производительности платформенного tracing-компонента (Фаза 17) на референсной лаборатории Gentoo (`192.168.100.70`, 8 vCPU, 32 GB RAM). Сессия охватила sign-off tier матрицы: M0–M10, queue saturation, M9 soak (60 мин).

**Ключевые выводы:**

1. **Нормальный режим (M5, sampling 0.1) не соответствует заявленным SLA** по CPU и памяти: медианная дельта к baseline M0 составляет **+48,1% CPU** и **+25,4% RSS** при бюджетах **3%** и **10%** соответственно. Это системное отклонение, а не погрешность измерения.

2. **Поведение в деградированных условиях (M6, M8a/b/c) соответствует ожиданиям модели устойчивости**: прирост HTTP p99 относительно M0 укладывается в **0,2–0,34 мс** (бюджет +50 мс); процесс завершался с exit code 0, OOM не зафиксирован.

3. **Soak-тест M9 (60 мин) не выявил признаков утечки памяти**: линейный наклон RSS **0,30 MB/мин** (бюджет ≤ 1,0 MB/мин).

4. **Квалификация референсной среды измерений — пограничная**: калибровочный прогон M0/M0 показал **1,77%** относительного дрейфа CPU при допустимом пороге **1,0%**. Среда пригодна для качественной оценки и сравнительного анализа, но **не обеспечивает статистическую достоверность для жёсткого 3%-SLA** без дополнительных мер (изоляция CPU, увеличение числа прогонов, повторная калибровка).

5. **Release gate `performanceReleaseGate` остаётся в состоянии FAIL**: все hard-бюджеты в `performance-budgets.yaml` — `PENDING`; evidence не привязан.

**Рекомендуемое решение комитета:** не допускать компонент к production sign-off в конфигурации M5 (FULL + ratio 0.1) до устранения корневой причины overhead **или** формального пересмотра SLA с оформлением waiver по governance-процедуре ADR.

## 2. Цель, объём и методология

### 2.1. Цель

Подтвердить соответствие платформенного tracing runtime требованиям `Traces Requests.txt`, переведённым в измеримые бюджеты (ADR-performance-model, Решение 1):

- CPU overhead (normal mode): **Δ < 3%** vs M0;
- Memory overhead (normal mode): **Δ RSS < 10%** vs M0;
- устойчивость при деградации Collector и bounded queue;
- отсутствие утечек на длительном горизонте.

### 2.2. Метод измерения нагрузки

Применён **open-model** генератор (k6 `constant-arrival-rate`). Closed-model намеренно не использовался: при деградации SUT closed-model снижает фактический RPS и маскирует рост латентности (coordinated omission). Open-model фиксирует целевой arrival rate независимо от времени отклика.

| Параметр | Значение |
|----------|----------|
| Целевой arrival rate | **300 req/s** (см. §3.2) |
| Warmup | 2 мин (не входит в оценку SLA) |
| Steady state | 10 мин (M9 — 60 мин) |
| Прогонов на конфигурацию | 3 (медиана валидных) |
| Validity gate | `dropped_iterations = 0`, k6 exit 0, полный steady window |

### 2.3. Baseline и протокол сравнения

- **Baseline M0:** прогон `2026-06-10_235521/m0` (без агента и extension).
- Все дельты CPU/RSS/p99 рассчитаны относительно этого baseline в рамках одной сессии на одном хосте.
- Метрики процесса: CPU utilization (нормализация per-core), RSS (`VmRSS`), линейный наклон RSS, HTTP p99 (k6).

## 3. Референсная измерительная среда

### 3.1. Конфигурация

| Параметр | Значение |
|----------|----------|
| Host ID | `gentoo-192-168-100-70` |
| Docker endpoint | `tcp://192.168.100.70:2375` |
| CPU | 8 cores |
| RAM | 32 GB |
| Kernel | 6.12.31-gentoo |
| Топология | SUT + OTel Collector + k6 (docker-compose.perf.yml) |

### 3.2. Квалификация среды (калибровка M0/M0)

Перед sign-off tier выполнена процедура калибровки измерительной среды: два последовательных прогона M0 без изменения конфигурации. Сравнивается относительный дрейф `process_cpu_delta_pct_per_core` между идентичными прогонами.

| Показатель | Результат | Порог ADR |
|------------|-----------|-----------|
| Δ CPU (M0 run A vs M0 run B) | **1,77%** | ≤ 1,0% |

**Интерпретация:** референсная среда **не прошла формальную квалификацию** для верификации 3%-бюджета. Фоновая вариативность потребления CPU (планировщик, соседние контейнеры, thermal throttling, сетевой стек remote Docker) превышает допустимую погрешность. Это не отменяет наблюдаемый порядок величины отклонения M5 (+48%), но требует:

- либо усиления изоляции (cpuset pinning, dedicated host),
- либо увеличения `runsPerConfig` до 5 при CV(M0) > 5%,
- либо пересмотра жёсткости порога калибровки на уровне комитета.

### 3.3. Операционный лимит нагрузки

Плановый rate **500 req/s** привёл к OOM генератора нагрузки (k6 exit 137) и нестабильности SUT на данной конфигурации 8c/32GB. Официальный rate сессии зафиксирован на **300 req/s** (`referenceLab.hardwareFingerprint.officialRateRps`). Сравнение с целевым production RPS требует экстраполяции или повторения на масштабированном стенде.

## 5. Результаты по сценариям

### 5.1. Baseline M0 (без tracing)

Референсный прогон `2026-06-10_235521`:

| Метрика | Значение |
|---------|----------|
| CPU utilization (per-core norm.) | 8,32% |
| RSS avg | 509,7 MB |
| RSS slope | 0,98 MB/мин |
| HTTP p99 | 1,14 ms |
| S1 startup (container → health 200) | 5 208 ms |

### 5.2. Декомпозиция стоимости (M4 — MINIMAL extension)

Медиана по 3 валидным прогонам (Δ vs M0):

| Метрика | Медиана |
|---------|---------|
| Δ CPU | **+22,7%** |
| Δ RSS | **+21,3%** |
| Δ p99 | +0,39 ms |

Вывод: даже каркас агента + MINIMAL extension потребляет существенную долю бюджета. Основной вклад в overhead формируется до включения FULL-функциональности и scrubbing.

### 5.3. Normal mode — M5 (FULL, ratio 0.1) — **критический раздел**

Медиана по **2 валидным** прогонам (Δ vs M0):

| Метрика | Медиана | Бюджет | Вердикт |
|---------|---------|--------|---------|
| Δ CPU | **+48,1%** | < 3,0% | **FAIL** |
| Δ RSS | **+25,4%** | < 10,0% | **FAIL** |
| Δ p99 | +0,38 ms | — (evidence) | Приемлемо |
| RSS slope | 4,09 MB/мин | — | Требует JFR/GC-анализа |

Абсолютные значения (медианный валидный прогон `2026-06-11_012235`):

| | M0 | M5 | Δ |
|--|----|----|---|
| CPU util. | 8,32% | 12,46% | +4,14 п.п. |
| RSS avg | 509,7 MB | 635,9 MB | +126 MB |
| p99 | 1,14 ms | 1,51 ms | +0,37 ms |

**Интерпретация:** при штатной production-конфигурации (FULL extension, sampling 10%) компонент увеличивает потребление CPU процесса приблизительно в **1,5 раза** и RSS — на **четверть**. Отклонение на порядок превышает норматив. При текущих бюджетах конфигурация M5 **не может быть рекомендована** для обязательного включения tracing на всех сервисах.

### 5.4. Worst case — M5w (FULL, ratio 1.0) — evidence

Единичный валидный прогон:

| Метрика | Значение |
|---------|----------|
| Δ CPU | **+114,7%** |
| Δ RSS | +22,4% |
| Δ p99 | +0,48 ms |

Режим ratio 1.0 предназначен для диагностики, не для штатного SLA. Требуется отдельное решение комитета по бюджету worst case (`m5w-worst-case`, tier: evidence).

### 5.5. Деградированные режимы (M6, M8a, M8b, M8c)

Медиана Δ vs M0 по 3 валидным прогонам:

| Сценарий | Условие | Δ CPU | Δ RSS | Δ p99 | OOM / exit≠0 |
|----------|---------|-------|-------|-------|--------------|
| M6 | Collector unreachable | +52,9% | +27,2% | +0,27 ms | Нет |
| M8a | Collector slow (+200 ms) | +35,2% | +19,5% | +0,20 ms | Нет |
| M8b | Collector 429/503 | +33,7% | +20,0% | +0,31 ms | Нет |
| M8c | Collector flaky | +37,6% | +21,8% | +0,34 ms | Нет |

| Гейт | Бюджет | Результат |
|------|--------|-----------|
| `degraded-p99` | Δ p99 ≤ 50 ms | **PASS** (все < 1 ms) |
| `degraded-no-oom` | 0 | **PASS** (exit 0, без OOM) |

**Интерпретация:** инварианты устойчивости P1/P3 (ADR) подтверждены на макро-уровне: деградация инфраструктуры не приводит к катастрофическому росту латентности HTTP и не роняет процесс. Рост CPU/RSS в деградированных режимах ожидаем (ретраи, queue pressure) и не оценивается теми же бюджетами, что M5.

Гейт `app-thread-no-export-wait` (JFR ThreadPark на exporter path для M6/M8a) **требует отдельного разбора JFR-артефактов** — автоматическая верификация в `summary.json` не выполнена.

### 5.6. Динамическая конфигурация — M10

Медиана по 3 валидным прогонам:

| Метрика | Медиана |
|---------|---------|
| Δ CPU | +45,0% |
| Δ RSS | +21,0% |
| Δ p99 | +0,27 ms |

Гейт `dynconf-no-stall` (p99 spike during reload ≤ 50 ms) требует анализа `reload-sequence.csv` и покомпонентных метрик в окне reload. Макро-уровень не показывает аномалий.

### 5.7. Queue saturation

| Ветка | Δ CPU (медиана) | Δ p99 | Δ RSS |
|-------|-----------------|-------|-------|
| DROP_OLDEST (`queue-saturation`) | +111,9% | +1,74 ms | +20,9% |
| UPSTREAM (`queue-saturation-upstream`) | +117,0% | +1,43 ms | +22,7% |

Гейт `queue-bounded` (queue size ≤ capacity) требует верификации по метрикам `droppedOverflow` / JMX в артефактах прогона. Макро-метрики подтверждают устойчивость процесса (exit 0, без OOM).

### 5.8. Soak — M9 (60 мин, FULL, ratio 0.1)

Прогон `2026-06-11_054302`:

| Метрика | Значение | Бюджет | Вердикт |
|---------|----------|--------|---------|
| RSS slope | **0,30 MB/мин** | ≤ 1,0 MB/мин | **PASS** |
| Δ CPU (vs M0) | +41,0% | — | Стабильный overhead, не дрейф |
| Δ RSS (vs M0) | +29,3% | — | Плато, не накопление |
| Δ p99 | −0,10 ms | — | Без деградации латентности |
| Steady samples | 698 (60 мин window) | — | Полное окно |

**Интерпретация:** на горизонте 60 минут утечка памяти не обнаружена. Абсолютный overhead остаётся повышенным, но **не нарастает** — это принципиально отличает ситуацию от leak-сценария.

## 7. Методологические и инфраструктурные находки

В ходе сессии выявлены и устранены дефекты инфраструктуры, не влияющие на интерпретацию основного вывода (M5 FAIL), но важные для воспроизводимости:

| Находка | Влияние | Митигация |
|---------|---------|-----------|
| High-cardinality URL в k6 (`/api/orders/{id}`) | `dropped_iterations`, инвалидация прогонов | `name`-теги в `steady-state.js` |
| PowerShell UA блокируется на remote API | Сбой M8a toxiproxy setup | Явный `User-Agent` |
| Ложный запуск `backpressure` для non-M8b | Прерывание resume | `PERF_BACKPRESSURE` только в `m8b.env` |
| Docker compose stderr → NativeCommandError | Ложные падения раннера | `2>&1` в `Invoke-Compose` |
| Rate 500 rps → OOM k6/SUT | Несопоставимость с планом | Официальный rate 300 rps |

## 8. Оценка рисков

| Риск | Вероятность | Влияние | Комментарий |
|------|-------------|---------|-------------|
| Обязательное включение FULL tracing на всех сервисах | Высокая (текущий default) | **Высокое** | +48% CPU → прямое влияние на cost of compute и capacity planning |
| Ложное PASS при повторении на «чистой» среде | Средняя | Среднее | Даже с идеальной калибровкой (1%) отклонение 48% не объясняется погрешностью |
| Неполная верификация JFR-гейтов | Средняя | Среднее | Макро PASS не заменяет micro-invariant ThreadPark |
| Экстраполяция 300 → production RPS | Высокая | Среднее | Требуется стенд production-класса |
| Единичный M9 / M5w прогон | Средняя | Низкое | Для soak критичен slope, не N; для M5w — evidence |

## 9. Решения, требуемые от архитектурного комитета

### 9.1. Обязательные (блокируют sign-off)

## 10. План действий (roadmap)

### Фаза A — Закрытие evidence gap (1–2 недели)

1. **JFR-анализ** прогонов M5, M6, M8a: flamegraph, ThreadPark, alloc rate → `docs/tracing/perf-results/2026-06-10_official/analysis/`.
2. **Повтор M5 ×3** на квалифицированной среде (cpuset pinning, N=5) для статистически значимой медианы.
3. **Закрытие pending gates**: `queue-bounded`, `dynconf-no-stall` — автоматизация в `analyze-perf-run.ps1`.
4. **JMH baseline** на той же hardware fingerprint → `jmhCompareBaseline`.

### Фаза B — Optimization program (при решении D1-A)

Приоритеты по ADR-scrubbing-cost и JFR:

1. Профиль `performance-default` vs `strict` — снижение scrubbing cost на hot path.
2. Декомпозиция M4→M5: изоляция стоимости scrubbing, export pipeline, agent instrumentation.
3. Целевые JMH-регрессии: `CompositeSampler`, `ScrubbingEngine`, `QueueOffer`.

Критерий выхода: M5 медиана Δ CPU < 3%, Δ RSS < 10% на квалифицированной среде, 3 валидных прогона.

### Фаза C — Governance closure

1. Обновить `performance-budgets.yaml`: `status: PASS | WAIVER` + `evidence:` ссылки.
2. Прогнать `./gradlew :platform-tracing-bench:performanceReleaseGate` → green.
3. Обновить `docs/SUPPORTED.md` pre-release checklist.
4. Зафиксировать решения D1–D6 в ADR-amendment или minutes комитета.
