# Clean Architecture Purist

Review Perspective: **Clean Architecture Purist**

## 1. Baseline Interpretation

The package is already correctly isolated from infrastructure: it is a pure-Java, OTel-free policy engine, and ArchUnit enforces that `core.sampling..` must not depend on OTel, Spring, or JMX. The runtime hot path is also clean in shape: OTel input is translated in `SamplingPolicyOtelAdapter`, evaluated by `SamplingPolicyEngine`, and translated back to OTel `SamplingResult` outside the core package.[^1]

However, the domain boundary is only partially clean. `SamplingPolicySnapshot` is both immutable runtime policy state and a configuration compiler/normalizer, while strict validation lives outside the core in `SamplerPolicyUpdate`, and duplicated normalization exists between `SamplerState` and `SamplingPolicySnapshot`. The current core therefore mixes domain model and configuration-compilation concerns, even though configuration “does not enter this package directly.”[^1]

The rule engine is explicit and deterministic: seven stateless rules are evaluated in a normative hard-coded order, `DefaultRatioPolicyRule` ensures the production chain never abstains, route-ratio semantics are longest-prefix-first, and `TraceIdRatioDecision` is the parity-critical probability algorithm. Those behaviors are stable domain semantics and should remain intact even if the architecture is cleaned up.[^1]

## 2. Main Architectural Problems

1. **`SamplingPolicySnapshot` mixes domain state with compilation concerns.** It is simultaneously the runtime policy object and the place where normalization, filtering, and partial validation happen through `fromConfiguration(...)`.[^1]
2. **Validation ownership is split across layers.** `SamplerPolicyUpdate` performs strict runtime validation in otel-extension, while core silently skips invalid route entries and only validates `defaultRatio` bounds in the snapshot constructor.[^1]
3. **Normalization is duplicated across adapter and domain-adjacent code.** `SamplerState.normalize*` and `SamplingPolicySnapshot.normalize*` duplicate logic, which is exactly the kind of drift Clean Architecture tries to prevent.[^1]
4. **The domain surface exposes implementation details.** All seven rule classes are public, `RouteRatioPrefix` is public despite being described as a compiled route prefix model, and the intended SPI boundary is unclear.[^1]
5. **Domain semantics are encoded partly in factories and utilities instead of explicit domain types.** `productionEngine()` hard-codes the normative rule order, `SamplingPolicyRuleNames` mixes rule names and metric concerns, and `TraceIdRatioDecision` is important but hidden and under-tested.[^1]
6. **The abstain protocol is not fully modeled.** Rules return `null` to abstain rather than an explicit domain result, which weakens the purity of the rule contract.[^1]

## 3. Recommended Architecture

I recommend a **clean domain core with explicit configuration compilation and validation layers**, while preserving all current runtime semantics. The domain should be centered on immutable types representing **request**, **decision**, **policy**, and **rule evaluation**, while all parsing, normalization, validation, and snapshot compilation should move out of `SamplingPolicySnapshot` into explicit collaborators.[^1]

In concrete terms, the architecture should be split into:

- **Domain model**: `SamplingPolicyRequest`, `SamplingPolicyDecision`, `SamplingPolicyReason`, `ParentContextState`, `SamplingPolicySnapshot`, and route-ratio value types.[^1]
- **Policy evaluation**: `SamplingPolicyEngine` plus rules, ideally with an explicit abstain result rather than `null`.[^1]
- **Configuration compilation**: a dedicated compiler that turns normalized config into `SamplingPolicySnapshot`, owning longest-prefix sorting and structural normalization.[^1]
- **Validation**: a dedicated validator owning domain invariants for policy configuration, instead of scattering them between otel-extension and core.[^1]
- **Adapters**: OTel request/result mapping and runtime control-plane update logic remain outside the core, where they already belong.[^1]

This is not a call for heavy abstraction. It is a call to stop hiding real concepts inside one overloaded class. The clean architecture move here is to make the existing conceptual stages explicit, not to invent a framework.

## 4. Classes to Keep

- **`SamplingPolicyRequest`** should remain mostly unchanged because it is already a clean, immutable input model decoupled from OTel.[^1]
- **`SamplingPolicyDecision`** should remain mostly unchanged because it is a proper domain result with invariant checks and factory methods.[^1]
- **`SamplingPolicyReason`** should remain because it represents stable decision semantics and bridges to platform reason codes.[^1]
- **`ParentContextState`** should remain because it is a compact domain-level representation of parent sampling state without OTel leakage.[^1]
- **`SamplingPolicyEngine.evaluate(request, snapshot)`** should remain conceptually unchanged because it is already a pure domain function.[^1]
- **Individual rule classes** should remain as the main units of policy behavior, because the inventory already shows they are isolated, stateless, and high-confidence refactoring seams.[^1]


## 5. Classes to Split / Rename / Move

- **`SamplingPolicySnapshot`** must be split in responsibility. It should remain the immutable compiled policy, but `fromConfiguration(...)` and normalization logic should move to a dedicated compiler.[^1]
- **`SamplingPolicyRule`** should be revised so abstain is an explicit result instead of `null`; this is a domain purity issue, not just a style issue.[^1]
- **`SamplingPolicyEngine.productionEngine()`** should move into a dedicated assembly/factory type so rule ordering is explicit policy assembly, not embedded runtime behavior.[^1]
- **`RouteRatioPrefix`** should be reconsidered as a domain value type with validation or moved to a more explicit `model`/`policy` location; today it is public but weakly modeled.[^1]
- **`SamplingPolicyRuleNames`** should move out of the rule domain if it primarily exists to centralize winning-rule strings; that concern is secondary metadata, not core policy behavior.[^1]
- **`TraceIdRatioDecision`** should move into a clearly named domain service or internal probability policy helper with direct tests, because it is too important to remain an opaque utility.[^1]


## 6. New Types to Introduce

- **`SamplingPolicyConfig`** — immutable domain-facing policy configuration before compilation; owns enabled flag, dropped routes, force values, route ratios, default ratio, and version.[^1]
- **`SamplingPolicyConfigValidator`** — validates config invariants consistently, replacing split ownership between `SamplerPolicyUpdate` and snapshot construction.[^1]
- **`SamplingPolicyCompiler`** — compiles `SamplingPolicyConfig` into `SamplingPolicySnapshot`, owning normalization and longest-prefix-first ordering.[^1]
- **`SamplingPolicyRuleResult`** — explicit result type for rule evaluation, eliminating `null` abstain and making rule contracts total.[^1]
- **`SamplingPolicyAssembly`** or **`ProductionSamplingPolicy`** — creates the normative production rule chain instead of embedding that assembly in `SamplingPolicyEngine.productionEngine()`.[^1]
- **`ProbabilitySamplerPolicy`** or similar internal helper — explicit home for the current `TraceIdRatioDecision` algorithm and tests.[^1]

## 8. Migration Plan

1. **Phase 1: Freeze current behavior with characterization tests.** Lock down rule order, route-prefix ordering, ratio parity, abstain/fallback semantics, and snapshot normalization before restructuring anything.[^1]
2. **Phase 2: Extract compilation without changing behavior.** Introduce `SamplingPolicyConfig` and `SamplingPolicyCompiler`, then make `SamplingPolicySnapshot.fromConfiguration(...)` delegate to the compiler as a compatibility facade.[^1]
3. **Phase 3: Centralize validation.** Introduce `SamplingPolicyConfigValidator` and migrate strict validation logic toward a single domain validator, while preserving current outward behavior for otel-extension callers.[^1]
4. **Phase 4: Purify rule contracts.** Introduce `SamplingPolicyRuleResult` and adapt `SamplingPolicyEngine` to stop using `null` abstain internally, while preserving external `SamplingPolicyDecision.abstain()` behavior.[^1]
5. **Phase 5: Separate assembly from evaluation.** Move `productionEngine()` construction into `ProductionSamplingPolicy` or similar, keeping the same normative rule order.[^1]
6. **Phase 6: Tighten boundaries.** Reassess which rule classes and value types should remain public once true external consumers are verified.[^1]

## 9. Tests Required Before Refactoring

- **`QaTracePolicyRuleTest`** because it currently has no dedicated core test coverage.[^1]
- **`TraceIdRatioDecisionTest`** covering null/short trace IDs, 0/1 boundaries, multiple fractional probabilities, and parity-oriented fixtures.[^1]
- **`SamplingPolicySnapshot.fromConfiguration`** tests for silent skipping of invalid route entries, out-of-range default ratio, and copy/immutability guarantees.[^1]
- **Fractional ratio tests** for `DefaultRatioPolicyRule` and `RouteRatioPolicyRule`, not just 0 and 1 behavior.[^1]
- **`SamplingPolicyDecision` invariant tests** for all valid and invalid constructor combinations.[^1]
- **Engine characterization tests** proving production rule order and “never abstains in production” behavior remain unchanged.[^1]
- **Compiler equivalence tests** once the compiler exists: old `fromConfiguration(...)` output vs new compiled snapshot output must match exactly.[^1]
- **Validator contract tests** ensuring validation semantics are explicit and documented where current behavior is intentionally lenient or strict.[^1]

## 11. Objections to Your Own Recommendation

The strongest objection is that this recommendation may be too ambitious for a package of only about 17 production types, especially since the current architecture is already OTel-free and operationally stable. Clean Architecture can become over-segmentation if the team lacks ongoing pressure from new policies, new config sources, or new adapters.[^1]

A second objection is that unifying validation in the core may conflict with the current intentional split, where runtime input validation and last-known-good behavior live in otel-extension. If that split is deliberate for operational reasons, the cleanest domain design may not be the most pragmatic platform design.[^1]

A third objection is that replacing `null` abstain with an explicit result improves purity but adds migration work and may not materially improve runtime behavior. That change is justified architecturally, but it is not free.

## 12. Final Verdict

**Proceed only after tests.**

From a Clean Architecture perspective, the package is close to good design but stops short of a truly explicit domain model because compilation, normalization, validation, and evaluation are not cleanly separated. The right move is to proceed with a domain-purifying refactor centered on compiler/validator extraction and explicit rule contracts, but only after strengthening the missing tests and preserving the documented runtime semantics exactly.[^1]
