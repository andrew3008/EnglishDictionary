Review Perspective: **Adversarial Reviewer**

## 1. Baseline Interpretation

The package is already operationally stable: it is pure Java, OTel-free, and evaluates sampling through a fixed seven-rule chain over immutable snapshot/request objects. The production rule order is normative, `DefaultRatioPolicyRule` never abstains, and route ratios are sorted longest-prefix-first at snapshot compile time.[^1]

The integration story is intentionally split: `platform-tracing-otel-extension` builds `SamplingPolicyRequest`, compiles `SamplingPolicySnapshot`, and maps `SamplingPolicyDecision` back to OTel `SamplingResult`. Only the otel-extension imports the core package in production, and the module boundary explicitly forbids OTel/Spring/JMX dependencies in `core.sampling`.[^1]

That means most refactoring risk here is not “how to make it work,” but “how to avoid breaking a working contract while adding abstraction that the package may not need.” The inventory also calls out duplicated normalization, split validation ownership, public rule classes, an unused `RECORD_ONLY` enum value, and an untested `TraceIdRatioDecision` as the main pain points.[^1]

## 2. Main Architectural Problems

1. **Over-engineering risk is high.** The package is small, already immutable, and already properly separated from OTel/Spring; large rewrites can easily create more complexity than value.[^1]
2. **Behavioral contracts are fragile and easy to break.** Rule order, parity with OTel ratio sampling, and longest-prefix route matching are all behaviorally important and already normative.[^1]
3. **The current split validation/normalization is ugly, but a big redesign may be worse.** Duplicated normalization exists, yet the safe fix may be much smaller than a new compiler/validator stack.[^1]
4. **Public rule classes are ambiguous, but tightening them may be a compatibility trap.** The inventory does not prove nobody external uses them; visibility changes could break hidden consumers.[^1]
5. **`SamplingPolicySnapshot` already mixes runtime and config concerns, but splitting it may create a second source of truth.** That would be a classic refactor trap if done before exhaustive tests.[^1]
6. **`TraceIdRatioDecision` is a hidden correctness kernel.** Any move or rewrite here can silently desynchronize from OTel parity if not protected with direct tests.[^1]

## 3. Recommended Architecture

My recommendation is: **do not pursue a broad architecture redesign**. Keep the package shape mostly intact and apply only the smallest set of changes that remove obvious duplication and accidental exposure, while refusing to introduce new layers unless the tests prove a hard need.[^1]

The best-adversarial stance is to treat `core.sampling` as a **working contract**, not an architecture playground. That means preserving the current engine, current rules, current request/decision model, current production order, and current adapter boundary, while limiting refactoring to defensive extraction of normalization and ratio logic into internal helpers if—and only if—those helpers can be introduced without semantic drift.[^1]

## 4. Classes to Keep

- **`SamplingPolicyEngine`** should stay because the existing rule-chain behavior is explicit, tested, and already aligned with the production contract.[^1]
- **`SamplingPolicyRequest`** should stay because it is a clean adapter boundary type and already decoupled from OTel.[^1]
- **`SamplingPolicyDecision`** should stay because it is the correct typed outcome and already guards invariants.[^1]
- **`SamplingPolicyReason`** should stay because reason-code mapping is one of the few externally meaningful semantic bridges.[^1]
- **`ParentContextState`** should stay because it is tiny and stable, and any “cleaner” abstraction would likely be needless churn.[^1]
- **All seven rule classes** should stay for now because there is no evidence that a more abstract rule registry would improve the package more than it would complicate it.[^1]


## 5. Classes to Split / Rename / Move

- **`SamplingPolicySnapshot`** is the only class that clearly deserves caution: if you split it, do so only to isolate normalization helpers, not to introduce a broader config pipeline.[^1]
- **`TraceIdRatioDecision`** should remain exactly where it is unless a dedicated test suite proves a clean move is safe; it is too important to touch casually.[^1]
- **`SamplingPolicyRuleNames`** could be renamed or grouped internally, but only if that is purely cosmetic and does not affect behavior or visibility.[^1]
- **`SamplingPolicyDecisionType`** may eventually need cleanup because `RECORD_ONLY` appears unused, but removing or repurposing it before proving no external dependency exists would be reckless.[^1]
- **`SamplingPolicyEngine.productionEngine()`** should not be abstracted away unless the current hard-coded order is a real problem; right now it is a known-good contract, not a smell that must be “fixed.”[^1]


## 6. New Types to Introduce

- **`SnapshotNormalizer`** only, if and only if it is a direct extraction of existing code with no behavior changes.[^1]
- **`TraceIdRatioDecisionTest`** as a new test seam rather than a new abstraction first; the core algorithm needs protection before any structural change.[^1]
- **Possibly `SamplingPolicyEngineFactory`** only if the package later proves it truly needs alternate engine compositions; otherwise it is premature.[^1]

Anything beyond that—`SamplingPolicyConfig`, `SamplingPolicyValidator`, pipeline stages, registry layers—is likely to be a net negative unless you have a concrete downstream requirement that cannot be satisfied with the current model.[^1]

## 7. Package Structure

The safest target structure is almost no structural change:

```text
space.br1440.platform.tracing.core.sampling
├── existing model classes
├── existing engine
├── existing rules
└── internal
    ├── SnapshotNormalizer
    └── TraceIdRatioDecision
```

I would **not** split this into model/compiler/runtime/control subpackages yet. The package is not large enough to justify a deeper hierarchy, and the risk of creating “architecture for its own sake” is higher than the likely payoff.[^1]

## 8. Migration Plan

1. **Lock behavior with tests first.** No code movement until rule order, parity, invalid config handling, and fallback behavior are fully characterized.[^1]
2. **Extract only obviously repeated code.** If normalization can be moved into a helper without changing any public methods or semantics, do that next.[^1]
3. **Run all existing core and otel-extension tests.** Especially verify the parity and adapter characterization tests still pass unchanged.[^1]
4. **Delay any visibility tightening.** Do not make rules package-private or remove enum values until you have hard evidence no external consumer depends on them.[^1]
5. **Stop there unless you find a real bug.** If the package is stable after the safe extraction, do not “improve” it further just because the design space allows it.[^1]

## 9. Tests Required Before Refactoring

- `SamplingPolicyEngineTest` confirming production order and that the production engine never abstains.[^1]
- `SamplingPolicySnapshotTest` covering normalization, immutability, route sort order, and invalid route entry handling.[^1]
- `TraceIdRatioDecisionTest` covering short IDs, null IDs, 0/1 boundaries, and several fractional ratios.[^1]
- `QaTracePolicyRuleTest`, since the inventory says this is missing in core.[^1]
- `SamplingPolicyDecision` invariant tests for all valid and invalid combinations.[^1]
- `TraceIdRatioParityTest` and `SamplingPolicyDecisionOtelAdapterTest` in the otel-extension path.[^1]
- A regression test proving that `productionEngine()` rule order is unchanged and that longest-prefix-first route semantics still hold.[^1]

## 11. Objections to Your Own Recommendation

The obvious objection is that this is too timid: it leaves the duplicated normalization and mixed responsibilities in place longer than a purist would want. That is true, but adversarial review should prefer a tolerable smell over a risky redesign, especially when the current behavior is already stable and the package boundary is already clean.[^1]

Another objection is that keeping public rule classes and unused enum values preserves accidental API surface. Again, that is a real problem, but removing them blindly is exactly how you create hidden breakage in a library whose external usage is not fully enumerated.[^1]

Finally, this recommendation may under-address future control-plane growth. If the package starts absorbing new config sources or alternate sampling policies, the minimal approach may become insufficient and a broader refactor could then be justified.[^1]

## 12. Final Verdict

**Defer.**

This package does not currently justify a big architectural move, and the safest answer is to postpone major refactoring until tests prove a specific defect or maintenance pain that cannot be solved with a small internal extraction. If you do anything now, keep it to the smallest possible behavior-preserving cleanup and stop immediately if the change starts to look like a framework rather than a refactor.[^1]
