Review Perspective: **Conservative Maintainer**

## 1. Baseline Interpretation

The package is already a pure-Java, OTel-free sampling engine with a fixed production rule chain and immutable snapshot-based evaluation. Configuration enters from `platform-tracing-otel-extension` through `SamplerPolicyUpdate` and `SamplerState`, and the core package only consumes the compiled `SamplingPolicySnapshot` plus the per-span `SamplingPolicyRequest`.[^1]

The current behavior is already well characterized: production rule order is normative, `DefaultRatioPolicyRule` never abstains, route ratios are sorted longest-prefix-first at snapshot compile time, and `TraceIdRatioDecision` is the critical parity algorithm for probabilistic sampling. The core module also has 37 tests passing, and the inventory explicitly calls out that the biggest factual pain points are duplicated normalization, split validation ownership, a hard-coded production chain, an unused `RECORD_ONLY` decision type, and public rule classes that may be more visible than necessary.[^1]

For this perspective, the important point is that the package is already operationally sound, so any refactor should be minimal, behavior-preserving, and justified by high-confidence evidence only.[^1]

## 2. Main Architectural Problems

1. **Duplicated normalization between core and otel-extension.** `SamplerState` and `SamplingPolicySnapshot` both normalize drops, force-values, and route ratios, which creates drift risk.[^1]
2. **`SamplingPolicySnapshot` owns too much.** It is both the immutable runtime policy and the place where configuration is compiled and filtered, which makes the code harder to reason about and test.[^1]
3. **The production chain is hard-coded.** `SamplingPolicyEngine.productionEngine()` bakes in rule order, which is correct today but makes future changes more fragile.[^1]
4. **Public rule classes may expose more API than intended.** All seven rules are public final, but the inventory suggests they are all stateless implementation details in practice.[^1]
5. **`RECORD_ONLY` is dead surface area.** It exists in the enum but is never produced by any rule, so it adds confusion without current value.[^1]
6. **`TraceIdRatioDecision` is too opaque for a core correctness primitive.** It is package-private and lacks a dedicated test class, despite being central to trace-id parity.[^1]

## 3. Recommended Architecture

Keep the current architecture mostly intact, but make a **small, surgical cleanup**: extract shared normalization and trace-id ratio logic into explicit internal helpers, tighten the visibility of clearly internal types, and leave the core runtime contract unchanged.[^1]

The right conservative move is not a redesign; it is a cleanup of the most obvious duplication and visibility issues while preserving the existing engine, rules, snapshot semantics, and OTel adapter flow exactly as they are. In particular, do **not** introduce a new layered model/validator/compiler stack unless tests and maintenance pressure prove that the current split is no longer sustainable.[^1]

## 4. Classes to Keep

- **`SamplingPolicyEngine`** should stay mostly unchanged because its current role and production rule order are normative and already tested.[^1]
- **`SamplingPolicyRequest`** should remain unchanged because it is already the correct adapter-friendly input shape.[^1]
- **`SamplingPolicyDecision`** should remain unchanged because it is the stable decision contract and already validates its own invariants.[^1]
- **`SamplingPolicyReason`** should remain unchanged because it maps to stable reason codes used by the platform.[^1]
- **`ParentContextState`** should remain unchanged because it is small, explicit, and isolated from OTel APIs.[^1]
- **`SamplingPolicyRule`** should remain as-is for now because the engine already depends on it and the inventory shows current rules are stateless.[^1]


## 5. Classes to Split / Rename / Move

- **`SamplingPolicySnapshot`** should be split only enough to remove duplicated normalization logic; its public API should remain intact.[^1]
- **`SamplingPolicyEngine.productionEngine()`** should be left alone for now, but the chain definition should be isolated in a helper/factory if and only if that can be done without changing behavior.[^1]
- **`TraceIdRatioDecision`** should be kept package-private but moved into a clearly named internal utility location if package structure is adjusted.[^1]
- **`SamplingPolicyRuleNames`** should stay internal and may be grouped with other internals so it is clearly not API.[^1]
- **All seven rule classes** should stay public only if there is a real external need; otherwise they are candidates for package-private later, but not as a first-step change.[^1]
- **`SamplingPolicyDecisionType`** should be reconsidered, but only after confirming no downstream code depends on `RECORD_ONLY` as a reserved value.[^1]


## 6. New Types to Introduce

- **`SnapshotNormalizer`**: internal helper that centralizes drop-path, force-value, and route-ratio normalization so `SamplingPolicySnapshot` and `SamplerState` do not drift.[^1]
- **`TraceIdRatioSampler`** or similar: internal utility that encapsulates the current ratio algorithm behind a testable name.[^1]
- **`SamplingPolicyEngineFactory`** only if you want to isolate the hard-coded production chain without changing the public engine API; otherwise skip it for now.[^1]
- **`SamplingPolicySnapshotTest` additions** rather than new production types for most of the current pain points, because this perspective prefers minimizing code churn.[^1]


## 7. Package Structure

A conservative target structure would be:

```text
space.br1440.platform.tracing.core.sampling
├── SamplingPolicyRequest
├── SamplingPolicyDecision
├── SamplingPolicySnapshot
├── SamplingPolicyReason
├── ParentContextState
├── SamplingPolicyDecisionType
├── SamplingPolicyRule
├── SamplingPolicyEngine
├── KillSwitchPolicyRule
├── HardDropPolicyRule
├── ForceHeaderPolicyRule
├── QaTracePolicyRule
├── ParentSampledPolicyRule
├── RouteRatioPolicyRule
├── DefaultRatioPolicyRule
└── internal
    ├── SnapshotNormalizer
    ├── TraceIdRatioDecision
    └── SamplingPolicyRuleNames
```

This keeps package movement to a minimum while still making the internal helpers clearly non-API.[^1]

## 8. Migration Plan

1. **Add characterization tests first.** Lock down existing normalization, ratio parity, rule order, and edge-case behavior before changing code.[^1]
2. **Extract internal helpers without changing signatures.** Move normalization and ratio logic into internal helpers and have the current classes delegate to them.[^1]
3. **Re-run the full core and otel-extension tests.** Verify that no decision matrices changed and no adapter behavior shifted.[^1]
4. **Only then consider visibility tightening.** If external consumers are confirmed absent, make clearly internal helpers and maybe rule classes less visible in a later step.[^1]
5. **Defer larger cleanup.** Do not introduce a new validation/compiler layer unless the minimal extraction proves insufficient.[^1]

## 9. Tests Required Before Refactoring

- `SamplingPolicyEngineTest` for production order and non-abstain behavior.[^1]
- `SamplingPolicySnapshotTest` for normalization, immutability, route sorting, and invalid configuration handling.[^1]
- `TraceIdRatioDecisionTest` for short IDs, null IDs, 0/1 boundaries, and parity with OTel-style results.[^1]
- `QaTracePolicyRuleTest` because the inventory says it is missing in core.[^1]
- `SamplingPolicyDecision` invariant tests for all legal and illegal combinations.[^1]
- Adapter characterization tests in `platform-tracing-otel-extension`, especially `TraceIdRatioParityTest` and `SamplingPolicyDecisionOtelAdapterTest`.[^1]
- A regression test proving that `productionEngine()` preserves the exact current rule sequence.[^1]

## 11. Objections to Your Own Recommendation

The biggest weakness is that this approach may leave some structural debt in place, especially the fact that `SamplingPolicySnapshot` still mixes runtime policy and configuration compilation responsibilities. If the package is expected to evolve with more policies, more config sources, or more runtime control-plane behavior, the conservative cleanup may not go far enough.[^1]

There is also a risk that keeping all seven rules public preserves an accidental SPI longer than it should, which could make a later cleanup harder. But from a conservative standpoint, that risk is preferable to breaking compatibility prematurely.[^1]

## 12. Final Verdict

**Proceed only after tests.**

This package is already functioning well, and the safest refactoring is a small, behavior-preserving extraction of duplicated internal logic plus test hardening. Anything larger would be premature for a conservative maintainer and would raise migration risk without enough evidence of benefit.[^1]
