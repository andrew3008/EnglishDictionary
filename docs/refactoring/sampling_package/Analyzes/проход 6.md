Review Perspective: **Spring Boot Platform Reviewer**

## 1. Baseline Interpretation

The core package itself is intentionally Spring-free and JMX-free; the inventory says `core.sampling..` must not depend on Spring, OTel, or JMX. Configuration enters outside the core through `TracingProperties.Sampling` → `SamplingRuntimeConfig.from()` on the autoconfigure side, then through `SamplerPolicyUpdate` and `SamplerState` in `platform-tracing-otel-extension`, which compile a `SamplingPolicySnapshot` for the runtime sampler.[^1]

The runtime flow is already platform-oriented: Spring startup or JMX updates feed a `SamplerStateHolder`, which publishes a new snapshot; `CompositeSampler` then evaluates spans through the pure engine. The inventory also shows the key platform pain points: duplicated normalization between `SamplerState` and `SamplingPolicySnapshot`, split validation ownership, silent skipping of invalid route entries in core, and a hard-coded production engine chain.[^1]

For a Spring Boot platform lens, the important thing is that the configuration/control-plane boundary already exists, but it is not cleanly centralized. The autoconfigure module, JMX update path, and otel-extension all participate in policy compilation, which makes future platform behavior harder to evolve consistently.[^1]

## 2. Main Architectural Problems

1. **Configuration ownership is split across modules.** Spring binding lives in autoconfigure, validation lives partly in `SamplerPolicyUpdate`, and compilation/normalization is duplicated in `SamplerState` and `SamplingPolicySnapshot`.[^1]
2. **The control-plane contract is not explicit enough.** JMX updates, startup properties, and runtime state mutation all end up producing snapshots, but the lifecycle is spread across types rather than captured as a clear policy-update pipeline.[^1]
3. **Validation semantics are inconsistent.** Core silently skips invalid route entries while the extension layer does strict validation, which is risky for a platform feature that must behave predictably across startup and runtime changes.[^1]
4. **The production chain is fixed in code.** That is acceptable for a small library, but platform control-plane changes become harder when the chain cannot be assembled or observed independently.[^1]
5. **Spring-bound metadata is not clearly isolated from runtime policy.** The inventory shows `SamplingRouteRatiosWire` keeps insertion order for JMX wire format only, while actual matching semantics are different, which is exactly the kind of thing that causes platform drift.[^1]
6. **`RECORD_ONLY` looks like surplus surface area.** For a platform reviewer, unused states complicate configuration and operator understanding without helping current control-plane behavior.[^1]

## 3. Recommended Architecture

I would keep the core package untouched from a Spring perspective and **move all configuration lifecycle concerns into a single, explicit control-plane layer** in the platform modules. The right architecture is a clear pipeline: Spring Boot binding → normalized config object → validation → snapshot compilation → atomic runtime publication → pure core evaluation.[^1]

In practice, the platform side should own a first-class `SamplingPolicyUpdate`/`SamplingPolicyConfig` flow that is the single source of truth for startup properties, JMX updates, and runtime rebuilds. The core should only consume immutable compiled snapshots and should not be asked to understand whether a policy came from Spring Boot, JMX, or any other source.[^1]

That means the architectural move is not to “Spring-ify” the core, but to **standardize the platform control-plane boundary** around the core. The platform layer should normalize once, validate once, and compile once, instead of letting `SamplerState` and `SamplingPolicySnapshot` each perform overlapping work.[^1]

## 4. Classes to Keep

- **`SamplingPolicyEngine`** should remain unchanged because it is the pure runtime evaluator and should not absorb Spring concerns.[^1]
- **`SamplingPolicyRequest`** should remain unchanged because it is an immutable request model and already decoupled from configuration binding.[^1]
- **`SamplingPolicyDecision`** should remain unchanged because it is the stable runtime outcome that the platform publishes into OTel.[^1]
- **`SamplingPolicySnapshot`** should remain as the immutable compiled policy object, but not as the platform’s configuration compiler-of-record.[^1]
- **`ParentContextState`** and **`SamplingPolicyReason`** should remain because they are compact domain enums, not platform binding concerns.[^1]


## 5. Classes to Split / Rename / Move

- **`SamplerState`** should be split so it stops being both a normalized runtime state holder and a duplicate compiler/normalizer; platform control-plane logic should own that responsibility explicitly.[^1]
- **`SamplerPolicyUpdate`** should be moved closer to a dedicated control-plane package and become the canonical validation/transition entrypoint for both startup and JMX updates.[^1]
- **`SamplingRuntimeConfig`** should stay in autoconfigure, but it should produce a clearer intermediate config type instead of reaching too quickly toward runtime objects.[^1]
- **`SamplingRouteRatiosWire`** should remain a wire-format type, but its relationship to actual matching semantics should be made explicit to avoid operator confusion.[^1]
- **`SamplingPolicySnapshot.fromConfiguration(...)`** should be demoted to a compatibility path, not the main platform compilation path.[^1]
- **`SamplingPolicyDecisionType`** should be reviewed for removal of unused values if no current platform flow consumes them.[^1]


## 6. New Types to Introduce

- **`SamplingPolicyConfig`** — a normalized platform configuration record/DTO used across Spring binding, startup, and JMX update paths.[^1]
- **`SamplingPolicyConfigValidator`** — validates policy config once, with platform-friendly error reporting.[^1]
- **`SamplingPolicySnapshotCompiler`** — compiles validated config into `SamplingPolicySnapshot`, owning normalization and route sort order.[^1]
- **`SamplingPolicyControlPlane`** — centralizes update operations and atomic publication semantics for startup and runtime updates.[^1]
- **`SamplingPolicyUpdateResult`** — explicit status object for success/failure/validation messages, better suited for actuator/JMX feedback than silent skips.[^1]
- **`SamplingPolicyProperties`** or a similarly explicit Spring-facing binding model if the current `TracingProperties.Sampling` is too broad for the configuration surface.[^1]

## 8. Migration Plan

1. **Characterize the startup and runtime update paths.** Add tests for Spring binding, JMX updates, invalid input handling, and last-known-good behavior before changing code.[^1]
2. **Introduce a shared platform config model.** Make Spring binding and JMX update code produce the same `SamplingPolicyConfig` object, even if internal adapters still exist temporarily.[^1]
3. **Centralize validation.** Move strict validation into a single control-plane validator and stop letting core silently compensate for bad route entries.[^1]
4. **Add a snapshot compiler.** Make both startup and runtime updates call the compiler, then publish only compiled immutable snapshots to the sampler.[^1]
5. **Deprecate `fromConfiguration(...)` as a control-plane entrypoint.** Keep it only as a compatibility helper until downstream platform code fully migrates.[^1]
6. **Remove duplicate normalization from `SamplerState`.** Once the compiler is authoritative, the runtime state holder should only hold and publish compiled snapshots.[^1]

## 9. Tests Required Before Refactoring

- **Spring binding tests** for `TracingProperties.Sampling` and `SamplingRuntimeConfig.from()` covering defaults and edge cases.[^1]
- **JMX update tests** for `PlatformSamplingControl.updateSamplingPolicy(...)` validating success, rejection, and last-known-good behavior.[^1]
- **Control-plane equivalence tests** proving startup and runtime updates compile to the same snapshot for the same logical config.[^1]
- **Invalid config tests** for blank route keys, null ratios, out-of-range ratios, and force/drop array mismatches.[^1]
- **Route ordering tests** proving longest-prefix-first matching is preserved after compilation.[^1]
- **Atomic publication tests** for `SamplerStateHolder` to ensure no partial policy is ever visible during updates.[^1]
- **Regression tests** for adapter-level behavior through `CompositeSampler` after the platform pipeline is introduced.[^1]

## 11. Objections to Your Own Recommendation

The main objection is that this may be more structure than the current platform actually needs. The inventory shows the current flow works and is already mostly isolated from core, so adding a shared config/validator/compiler layer could be seen as formalizing a problem that is still manageable.[^1]

Another objection is that the split between autoconfigure and otel-extension is already intentional, and centralizing too much control-plane logic may blur module responsibilities instead of clarifying them.[^1]

Finally, if the platform is not expected to support richer runtime management, then moving from a simple snapshot path to a more explicit control-plane model may introduce ceremony without enough operational payoff.

## 12. Final Verdict

**Proceed only after tests.**

The architecture should evolve toward a single Spring/platform control-plane pipeline, but only after tests lock down startup binding, JMX updates, validation semantics, and snapshot equivalence. The core package itself should stay clean and Spring-free; the work belongs in the platform control-plane layer, not in the domain engine.[^1]
